## 视频通话系统 Bug 修复

### Bug：视频画面无法显示
- **根因**：`VideoCallSession.startVideoStreaming()` 中 `startReceiveThread()` 在 `capturing.set(true)` 之前被调用，接收线程检查 `capturing.get()` 为 `false` 直接退出 → 远端视频帧永远无法接收
- **修复**：将 `capturing.set(true)` 移到 `startVideoStreaming()` 中，在启动接收和采集两个子线程之前执行；从 `startCaptureThread()` 中移除冗余的 `capturing.set(true)`
- **文件**：`video/src/main/java/com/pmcl/video/VideoCallSession.java`

### 架构分析笔记
- 视频通话使用 ice4j (ICE/STUN) + JavaCV (FFmpegFrameGrabber) + JPEG/UDP 传输
- JPEG 质量 60，分辨率 640x480，15 FPS
- 无 TURN 中继（仅 STUN），对称 NAT 下可能失败
- `muted` 标志未在采集循环中生效（仅 UI 展现）
- `BufferedImage.toImageBitmap()` 每次做 PNG 编码→Skia 解码，15FPS 下有性能开销

## 第二轮修复：4 个致命 Bug 排查与修复

### Bug 2：Pending 队列死锁
- **根因**：`FriendManager.sendFriendRequest()` 当 IP 为空时，enqueue 到 pending 队列。但如果 TCP 客户端已通过 UDP 发现连上，队列在 `onConnected` 时已被清空，新加入的 action 永久卡死
- **修复**：在 `sendFriendRequest` 中，enqueue 前先检查 `activeClients` 是否已连通。新增 `flushIfConnected()` 辅助方法，在所有信令方法（sendCallInvite/sendCallAccept/sendCallReject/sendCallEnd/sendCallIceCandidate）中 enqueue 后立即 flush

### Bug 3：sendCallInvite 静默失败
- **根因**：`sendCallInvite` 返回 null 时 `FriendPage.kt` 无任何反馈，用户看到 "等待对方接听" 但邀请从未发出
- **修复**：`FriendPage.kt` 检查返回值，失败时显示诊断日志；`FriendManager.java` 添加详细错误日志（好友不存在/无网络地址）

### Bug 4：FFmpeg 原生库缺失
- **根因**：`libs.ffmpeg.macosx.arm64` 在 `libs.versions.toml` 中未指定 `classifier: macosx-arm64`，只拉取主 JAR 不含原生 .dylib
- **修复**：`video/build.gradle.kts` 改用 `implementation("org.bytedeco:ffmpeg:7.1-1.5.11:macosx-arm64")` 显式指定平台分类器
- **测试验证**：ICE 协商成功（12 个候选交换），IPv6 直连，摄像头采集已启动

### Bug 5：macOS 摄像头设备名和帧率
- **根因**：avfoundation 设备格式为 `video:audio`，原代码 `"0"` 只指定了一个设备；帧率 15 不被设备支持
- **修复**：设备名改为 `"0:none"`（视频设备0，无音频），帧率改为 30

## 第三轮修复：视频传输架构重构

### Bug 6：ICE 3 秒超时终止关闭 socket
- **根因**：视频接收线程和 ice4j STUN keepalive 线程共用同一 DatagramSocket，接收线程消耗了 STUN 响应，ice4j 判定连接超时 → Terminated → 关闭 socket
- **修复**：视频传输改用独立 DatagramSocket，通过信令协议交换视频端口（CallInvite/CallAccept 新增 videoPort 字段），ICE 完成后立即释放 agent

### Bug 7：JPEG 压缩质量参数未生效
- **根因**：`compressJpeg` 使用 `ImageIO.write` 忽略 quality 参数，默认质量产生超大帧被 MTU 检查静默丢弃
- **修复**：使用 `JPEGImageWriteParam.setCompressionQuality` 真正控制质量，超限自动降级重试（60→40→25）

### Bug 8：startVideoStreaming 覆盖预创建 socket
- **根因**：FriendPage 通过反射设置了 videoSocket，但 startVideoStreaming 又 `new DatagramSocket()` 覆盖，端口不匹配
- **修复**：检查 videoSocket 是否已存在再决定是否创建

### Bug 9：onRemoteVideoPort 时序死锁
- **根因**：onRemoteVideoPort 在 ICE 完成前被调用，此时 pendingIceRemote 为 null，方法直接返回，采集线程永不启动
- **修复**：onRemoteVideoPort 先存储端口，startVideoStreaming 中检查已存储端口并自动启动采集

### Bug 10（致命）：avfoundation 设备格式写反
- **根因**：`"none:0"` 在 avfoundation 格式 `video:audio` 中意为 video=none audio=0，只抓到音频流无视频流
- **修复**：改为 `"0:none"`（video=0, audio=none）
- **验证**：两端 640x480 30fps 视频帧正常交换，采集=320 发送=320 丢弃=0 接收=570
